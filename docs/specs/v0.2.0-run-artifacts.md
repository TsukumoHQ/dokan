# Spec — v0.2.0 "run artifacts" (input + output files)

Status: **draft** · Owner: dokan-core · Origin: owner (Loïc) + dokan-core, 2026-06-26 · CTO: GO to spec

## 1. Problem

Agents constantly need jobs to manipulate **local documents** — a PDF to parse, a
`.md` to lint, a dataset to crunch. dokan today has exactly one input channel:

```
DOKAN_INPUT = a JSON string in an env var
```

That caps at ~100 KB (ARG_MAX on the exec'd process). A real document doesn't fit.
The obvious reach — **bind-mounting a host directory into the container** — is wrong,
and not for a shallow reason:

- A mounted host dir is **mutable, uncaptured, and machine-local**. The job's output
  stops being a pure function of its declared input.
- That breaks dokan's two core guarantees: the **content-addressed cache** (the key
  hashes source+input+image, not the host FS → stale/incorrect hits) and the **signed
  receipt** (claims the output came from declared inputs → now a lie; not reproducible
  on another host).
- It also punctures the **container = isolation boundary** (jobs run arbitrary uploaded
  code). See `SECURITY.md`.

This is the same conclusion Bazel/Nix reach for hermetic builds: reproducibility requires
that *everything a job reads is a declared, content-addressed input* — no host leak.

## 2. Solution — upload files INTO the job, don't mount the host OUT

Snapshot the files into the run's input at trigger time. The uploaded bytes **are** the
input, so the job stays a pure function of its inputs and every guarantee holds.

```
upload_blob(bytes) -> handle            # content-addressed (blake3); dedup; re-upload = no-op
run_script(id, input, files:{ "rapport.md": <handle>, ... })
  -> dokan materializes the files READ-ONLY in /input/ before exec
  -> the blob hashes enter the cache key + the receipt
  -> determinism, cache, and a portable receipt all stay INTACT
```

Symmetric on output: the job writes to `/output/`, dokan captures the produced files as
**run artifacts** (the file-sized inverse of the small `::dokan:result:: {json}` line).

The host-access lives where it is legitimate — the **agent on the host** reads the doc and
attaches it; the container never pierces its wall.

### 2.1 Prior art (this is the convergent pattern, not a novel design)

| System | Input mechanism | Cap |
|---|---|---|
| **E2B** | `sandbox.files.write(path, content)` | 100 MB |
| **Modal** | filesystem API (read 100 MiB / write 16 MiB); **Volumes** = "upload once, reuse across sandboxes" | 100 MiB / 16 MiB |
| **OpenAI Code Interpreter** | upload → `/mnt/data`; pass `file_ids:[...]` (**= our handles**); outputs returned as `container_file_citation` (**= our `/output` capture**) | 4 g container |
| **Bazel / Nix** | content-addressed declared inputs, hermetic sandbox, **no host leak** | — |

`upload_blob → handle` is OpenAI's `file_ids`; `/output` capture is their
`container_file_citation`. The handle/CAS approach is what keeps it hermetic (Bazel).

## 3. API surface (MCP)

New / changed tools:

- **`upload_blob(bytes, filename?)` → `{ handle, size, sha }`**
  Content-addressed store. `handle` = the blake3 hex (or a short prefixed id over it).
  Re-uploading identical bytes returns the same handle, stores nothing new.
  Bytes arrive base64 in the MCP call (MCP is JSON); cap enforced server-side (§5).

- **`run_script(script_id, input?, files?)`** — `files` is a map
  `{ "<dest-name>": "<handle>" }`. Each handle is materialized read-only at
  `/input/<dest-name>` in the container. Unknown handle → loud error, run not created.

- **`run_flow(flow_id, input?, files?)`** — same `files` map; available to every step as
  `/input/` (flow-level artifacts), in addition to the existing `deps` wiring.

- **`get_run(run_id)` / `list_runs`** — gain an `artifacts` summary:
  `[{ name, handle, size }]` for output files captured from `/output/`.

- **`download_blob(handle)` → bytes** (base64 out) — fetch an input or output blob by
  handle. The operator/agent pulls an output file this way (OpenAI's
  `container_file_citation` download).

`upload_script` is **unchanged** — artifacts are a per-*run* concern, not per-script.

## 4. Container materialization

Integrates at the existing exec path (`src/exec.rs::exec_job`, `src/pool.rs::create_one`):

- **`/input/`** — a read-only mount populated before exec. Implementation: write the
  blobs to a per-run temp dir on the host, bind-mount it `:ro` into the container at
  `/input/`. (This host-side dir is dokan-controlled and ephemeral — NOT a user host
  path, so it does not reintroduce the hermeticity problem: its contents are exactly the
  declared, content-addressed input blobs.)
- **`/output/`** — a writable dir (tmpfs or bind) the job writes to. After exit, dokan
  reads it, stores each file as a CAS blob, records `{name, handle, size}` on the run.
  Cap total `/output` size (§5); over-cap → run fails loud, not silent-truncate.
- Hardening unchanged: `/input` ro, no extra caps, the job still runs network-isolated
  unless `network=true`.

Note: this is the **first legitimate bind-mount in dokan** — but of a *dokan-owned,
content-addressed, ephemeral* dir, not a user host path. The determinism argument is
preserved because the mounted bytes are precisely the declared inputs.

## 5. Storage (CAS)

- **Content-addressed**: key = blake3 of the bytes. Dedup is free (same doc across many
  runs = one blob). A `blobs` table: `sha PRIMARY KEY, bytes BYTEA, size BIGINT,
  created_at, last_used_at`.
- **v1 cap**: per-blob ≤ ~32 MiB, total `files` per run ≤ ~64 MiB (in line with E2B/Modal).
  Stored in Postgres `bytea`. Loud rejection over cap.
- **Later (v0.2.x)**: spill to a host blob-store (content-addressed files on disk) for
  larger payloads; the `blobs` row keeps metadata + a location pointer. Out of scope for
  v0.2.0.
- **GC**: blobs unreferenced by any run/script for N days are collectable (mirror the
  existing run/log GC). Reference = appears in a run's `files` (input) or `artifacts`
  (output).

## 6. Cache key + receipt

- The **cache key** already hashes (source + input + image digest). Extend it to include
  the **sorted list of input blob handles** (`name:sha`). Same script + same input + same
  files → cache hit. Change a file → new key → recompute. Correct + deterministic.
- The **receipt** records the input blob handles (and output artifact handles). The run's
  output is now provably a function of (source, input, image, input-blobs) — reproducible
  on any executor that can fetch those blobs by handle. Portability preserved.

## 7. Backward compatibility

- Fully additive. `files` is optional everywhere; a run with no `files` behaves exactly
  as today. No schema change to `scripts`; new `blobs` table + new columns on `runs`
  (`artifacts JSONB`), all additive `ADD COLUMN IF NOT EXISTS` / `CREATE TABLE IF NOT
  EXISTS`.
- No change to `DOKAN_INPUT` semantics. `/input` + `/output` are a new, parallel channel.

## 8. Phasing

- **v0.2.0**: input artifacts (`upload_blob`, `run_script files:`, `/input` ro mount,
  CAS, cache-key + receipt extension) — the lever for "agents manipulate local docs".
- **v0.2.1**: output artifacts (`/output` capture, `download_blob`, `get_run.artifacts`).
  Splittable if v0.2.0 is getting large; input is the higher-value half.
- **later**: host blob-store for >cap payloads; `run_flow files:` if not in v0.2.0.

## 9. Security

- Single-tenant trust model unchanged (`SECURITY.md`). Blobs are operator-scoped like
  secrets; no per-agent ACL in v1.
- `/input` is read-only → a job can't tamper with a shared input blob.
- Cap + loud-reject prevents a job filling the store via `/output`.
- `download_blob` returns operator data — same trust boundary as `get_script` source.

## 10. Open questions

1. **Handle format** — raw blake3 hex, or a prefixed id (`blob_<hex>`) for readability?
2. **Inline small files** — allow `files:{ "x.md": { data: "<base64>" } }` for tiny
   content without a separate `upload_blob` round-trip? (E2B/OpenAI both allow inline.)
   Lean yes — inline ≤ a few KB, handle for the rest.
3. **`/output` capture policy** — all files, or only those under a declared manifest?
   Lean "all files present in `/output` at exit", capped.
4. **Dedup eviction** — LRU on `last_used_at` vs pure age GC. Lean age GC for v1.
5. **Mount vs copy for `/input`** — bind-mount the per-run temp dir (cheap) vs copy bytes
   into the container (no host dir at all, fully OCI-portable). Bind-mount is simpler;
   copy is cleaner for a future remote executor. Lean bind-mount for v0.2.0, note the
   copy path for remote.

## 11. Why not the alternatives (recap)

- **Host bind-mount of a user path** — breaks determinism/cache/receipt + hermeticity.
  Kept only as a niche opt-in (`mount_ro`, `cache:false`) for live-host-scan jobs that
  explicitly forgo reproducibility; NOT the main path.
- **Stuff bytes into `DOKAN_INPUT`** — env var, ~100 KB cap, no dedup, pollutes the JSON
  input contract. Fine for tiny inline content (see Q2), not for documents.

### Implementation sketch (for the build phase, not part of approval)

- `src/db.rs`: `blobs` table + `put_blob`/`get_blob`/`gc_blobs`; `runs.artifacts JSONB`.
- `src/mcp.rs`: `upload_blob`, `download_blob` tools; `files` arg on `run_script`/`run_flow`.
- `src/exec.rs`: materialize `/input` (write temp dir + bind ro) before exec; capture
  `/output` after; thread input-blob handles into the cache key + receipt.
- `src/pool.rs`: `create_one` gains optional `(input_dir, output_dir)` bind mounts.
- tests: `run_with_input_file` (upload → run → job reads `/input/x` → succeeds, cache key
  reflects the blob), `output_artifact_captured`.
